1. 背景与目标结构优化
物品不需要数量限制。角色的背景与目标使用数组存储，例如"Background": ["...", "..."]，由后端系统根据指令自动按顺序写入或删除特定的条目。数组中的每一条内容必须是一个完全独立的事件或事实，绝不将同一件事情拆分为几条无意义的片段。

2. 战斗信息全景化
角色的战斗信息必须精细化，确保列出游戏中玩家或NPC实际需要用到的所有动作，包括近战攻击、远程攻击、戏法、核心法术以及特殊能力。这些动作需要详细列出伤害骰、作用效果以及使用条件。

3. fixed属性的手动干预
设定上带有下划线前缀的fixed字段（不可变字段），在面临特殊游戏事件（如角色升级带来的属性变动）时，应该是可以由玩家主动手动解除锁定并进行一次性修改的，修改完成后再重新锁定。

4. 隐藏信息的上帝视角与解密机制
传给AI模型的信息统一采用上帝视角，其中玩家角色不知情的部分使用[HIDDEN]标记包裹。在未来的开发中，需要开发出对应的函数接口，当玩家在游戏中探索发现该线索时，系统自动对该条目解除[HIDDEN]标记，使其转变为公开情报。

5. 角色态度的独立更新规则
为了让AI更准确地把握NPC性格，不仅要有态度分值和现状描述，还需要在动态属性中增加态度更新规则字段（attitude_update_rule）。该规则专门描述该角色个人的倾向、好恶，用以指导AI判断哪些行为会导致好感增加或下降。给LLM发送信息时，系统还会提供一个全局的关系判断总规则配合使用。

6. 装备栏计算与正则解析
设立规范的装备栏格式，包括主手、副手、防具等。开发对应的正则解析工具来提取武器和防具名称及数值，配合基础九维属性公式，实现HP、AC以及近战/远程伤害的动态自动计算，而不是写死。

7. 外貌数据的细化与分离
人物的外貌数据（_appearance）必须将身体面容（_Body_and_Face）与穿着（_Clothing）分离开来。其中_Body_and_Face必须详细且结构化地包含：面部气质、瞳色、头发（包含颜色、曲直、长短、发型）、三维、身高。所有这些生理数据需要严格参数化，以供后续可能的游戏逻辑（如伪装、魅力检定、体型检定）进行读取。

8. 全局状态（world.json）的更新规则与机制
全局状态（如全局叙事摘要、重大事件历史）并不是随着每一次玩家输入同步更新的，而是采取“满足触发条件后单次拉起”的策略。
更新机制如下：
(1) 触发阈值：后端判定发生了诸如某NPC死亡、某重要地点关闭、时间每经过7天，或AI在地点/角色结算的Analysis中输出了类似于 [MAJOR_EVENT_TRIGGER] 的标签。
(2) 调用注入：触发后，系统拉起一次专门的“全局总结”LLM调用。此次调用会将当前的 world.json 完整结构（包含其中的动态数组），以及自上次更新以来各个地点的最新记录注入给LLM。
(3) 准确更新：由于LLM在本次调用中看到的是包含所有过去记录的**完整数组结构**（即完整的当前层级数据），它完全不需要依赖长程对话记忆去回想之前记录了什么。它只需评估最新的日志，一旦确认需要添加事件，直接输出标准的 JSONPatch 指令（例如：{"op": "add", "path": "/dynamic/major_event_history/-", "value": "..."}），以此将最新的总结准确无误地追加至数组末尾。

9. JSONPatch 路径的"单一坐标系"原则
核心思想：注入给 LLM 的 JSON 文本，和后端用来执行 patch 寻址的 Python 对象，必须永远是同一个。

具体实现：build_context() 不再返回序列化后的字符串，而是返回一个"活的"Python 字典 game_state。后端把这个字典既用 json.dumps() 序列化后注入 prompt，又直接保留为 patch 的寻址根对象。LLM 读到的 JSON 结构和后端字典结构完全一一对应，因此 LLM 写出的任何路径（如 /characters_snapshot/0/dynamic/attitude_value）后端都能在同一个对象上直接寻址，无需任何"猜测"性的路由逻辑。

这个原则对未来扩展完全成立：无论 build_context() 内部构造多么复杂（多角色、多地点、多故事线、物品系统），只要规则始终是"注入什么就在什么上寻址"，路径坐标系就永远不会错位。

附加规则：所有你希望 LLM 能修改的数据必须出现在注入的字典里。未注入的数据（如出于 token 节省只传了摘要的角色）LLM 无法写出对应路径，也就无法更新，这是正确行为而非缺陷。system_prompt 和 TOOLS 定义中的路径示例，也必须始终与当前 build_context() 输出的结构保持同步。


10. JSON 数据结构设计原则：以_前缀代替 fixed/dynamic 分层
问题来源：最初的数据文件同时使用了两套机制区分可变与不可变字段：一是放入 fixed 或 dynamic 的嵌套对象，二是对字段名加 _ 前缀。两套机制冗余，且导致可变字段散落在不同层级，LLM 生成的 JSONPatch 路径因此频繁错位。

核心原则：彻底废除 fixed/dynamic 容器对象。所有字段一律写在实体根层级，仅以 _ 前缀作为唯一可变性标记：
- 带 _ 前缀的字段：不可变（如 _combat_info、_world_rules），AI 不能对其发出 JSONPatch 指令，后端也会主动拦截。
- 不带 _ 前缀的字段：可变（如 attitude_value、current_hp、description），AI 可以修改。

结果：所有可变字段均在各实体根层级，路径唯一确定，不存在层级歧义。即使实体内部还有深层嵌套的 _ 对象，由于全部带 _ 前缀，LLM 被告知永不写入，后端也有防护拦截，不会造成混乱。

附：apply_patch 在路径解析之前会先检查路径的每一段，若任意一段以 _ 开头则直接返回 [blocked] 拒绝操作。

11. apply_patch 的容错原则
apply_patch 只返回错误字符串，不抛出异常。单条 patch 失败不中断后续执行，所有错误写入 run_log.txt 日志文件。

12. 运行日志写入文件（run_log.txt）
Windows 终端关闭后输出永久丢失。解决方案：定义 write_log() 和 log_error() 函数，所有重要输出都同时追加写入 run_log.txt，保证永久可查。

13. build_context() 的扁平注入规则
构造 characters_snapshot 时只传无 _ 前缀字段，LLM 看到的字段集合与被允许修改的集合完全一致。对 locations 也进行 deepcopy，保持与 world_state、char_state 同等的内存隔离保护。
